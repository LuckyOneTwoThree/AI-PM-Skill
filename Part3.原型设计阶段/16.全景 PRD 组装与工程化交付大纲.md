# 模块十六：全景 PRD 组装与工程化交付大纲 (Comprehensive PRD & Engineering Handoff)

在这个阶段，原始 Prompt 极其专业且全面。但如果在实际操作中，如果把那一大段 Prompt 一次性发给大模型（如 GPT-4o 或 Claude 3.5），100%会遇到大模型的绝症： **“输出截断（Output Truncation）”与“严重偷懒（Laziness）”**。

因为一份包含图表、权限、功能、埋点、非功能性需求（NFR）的完整 PRD，其文字量远远超过了单次回答的 Token 吐出上限。大模型为了强行在一轮对话里写完，会把极其重要的功能逻辑一笔带过。

为了获得 **“能直接丢给研发、且研发挑不出刺的工业级交付文档”**，我们必须采取 **“模块化分块渲染（Chunking Strategy）”**。我将其重构为 **4 个模块化装配步骤 + 1 个研发拒收预警审查**。

---

**👤 AI 角色定位：** 极度严谨的技术型产品总监 (Technical PM) / 研发效能专家 / 资深 QA 工程师

**🎯 核心目标：** 消除一切“应该、可能、大概”等模糊词汇。将前 15 个模块的所有推演，浇筑成没有歧义、带有强追溯链路、且研发可以直接照着敲代码的“契约合同”。

**📥 前置输入（变量）：** `[模块八的逻辑图]`、`[模块九的字段与ID]`、`[模块十一的页面骨架]`、`[模块十四的埋点表]`

---

## 🟢 步骤一：PRD 骨架搭建与全局约束锁定 (PRD Skeleton & NFRs)
*操作目的：先搭台，再唱戏。把全局权限矩阵和最容易被忽略的“非功能性需求（NFR）”以极其量化的方式钉死。*

- **操作方式：** 为大模型注入全局背景信息。
- **提示词模板：**
  ```text
  你现在是负责本项目研发交付的技术型产品专家。我们要开始撰写终局 PRD，为了防止遗漏，我们将分模块进行。
  首先，请基于《全局共识文档 (One-Pager)》，为我生成 PRD 的【全局配置大纲与非功能约束】部分。
  
  请使用 Markdown 严格输出以下模块：
  1. 需求背景与术语表 (Glossary)：一句话重申商业目标。并列出本项目中 3-5 个容易产生歧义的专有名词，给出绝对清晰的定义。
  2. RBAC 全局角色权限矩阵：用表格列出系统涉及的所有角色（如：未登录访客、普通会员、管理员），并打勾标记他们对核心模块的【增/删/改/查 (CRUD)】权限。
  3. 量化非功能性需求 (NFR - Non-Functional Requirements)：
     - 【防含糊红线】：严禁使用“响应要快”、“系统要安全”等废话。
     - 性能约束：核心接口响应时间必须小于多少毫秒？高并发峰值预估 TPS 是多少？
     - 兼容性约束：明确标出最低适配版本（如 iOS 14+ / Chrome 90+ / 微信 8.0+）。
     - 安全约束：指出哪些接口必须加上防重放、防刷限流机制。
  4. 变更履历表框架：生成一个标准的版本变更表格模板（含版本号、修订内容、修订人、日期）。
  ```
- **产出物：** 术语字典、RBAC 角色权限矩阵表、量化非功能需求 (NFR) 清单、PRD 版本基建模板。

---

## 🟢 步骤二：单模块功能深度刻画与追溯渲染 (Deep-Dive Functional Spec)
*操作目的：这是防止大模型偷懒的核心手段。不要让它一次写完所有功能，而是“按模块”一个个榨干细节。*

- **操作方式：** 指定你要让 AI 写的**某一个特定功能模块**（例如：“现在，请专门帮我撰写【订单支付模块】的详细需求”），配合原型截图（如果有）一起投喂。
- **提示词模板：**
  ```text
  现在，请聚焦于【填写具体模块名称，如：购物车结算模块】。
  请将该模块之前推演的所有逻辑、字段、UI、埋点进行“组装缝合”。这部分内容是后端工程师和前端工程师联调的唯一依据。
  
  针对该模块下的每一个功能点（L3级别），请按以下结构输出：
  1. 身份铭牌：功能名称 + [唯一功能 ID 映射] + [关联 User Story ID]。
  2. Mermaid 状态转换图：如果该功能涉及状态流转（如订单状态），请插入一幅局部的 Mermaid State Diagram。
  3. 界面与交互流 (Frontend Spec)：
     - 默认状态是什么？（如：提交按钮置灰）。
     - 触发条件是什么？前端有哪些强校验规则？（如：密码必须包含大小写，否则实时爆红）。
  4. 后端黑盒逻辑 (Backend Rules)：
     - 点击提交后，后端需要执行的核心逻辑步骤是什么？
     - 该功能依赖的【模块九：底层字段】是哪些？
  5. 异常预案与埋点 (Exceptions & Tracking)：
     - 列出 2 个该节点最容易发生的异常，并写明前端展示什么报错 Toast。
     - 关联对应的【模块十四：核心埋点事件 ID】。
  ```
- **产出物：** 单模块的“All-in-One”极度详尽需求文档（含追溯ID、前后端交互规则、Mermaid局部状态图、异常预案）。*（注：重复此步骤直到所有模块写完）*

## 🟢 步骤三：极恶测试用例生成与 BDD 验收矩阵 (QA Edge-Case Matrix)
*操作目的：如果 PRD 是进攻，测试用例就是防守。不要让 AI 泛泛而谈，让它用 BDD (行为驱动开发) 格式写出能直接执行的自动化脚本雏形。*

- **提示词模板：**
  ```text
  开发看完了 PRD，现在请切换为“绝不留情面”的高级 QA 测试工程师。基于刚刚生成的 PRD，编写该模块的测试用例。
  
  请用 Markdown 表格输出测试矩阵，必须包含以下极高密度的内容：
  1. 用例 ID 与 用例类型：（类型需区分：Happy Path 流程测试 / 边界值测试 / 接口异常测试 / 并发安全测试）。
  2. BDD 测试步骤 (Given-When-Then)：
     - Given (前置条件)：如“账户余额为 0 且处于未登录状态”。
     - When (测试动作)：如“用户狂点 5 次立即支付按钮”。
     - Then (预期结果)：如“系统拦截前 4 次请求，第 5 次跳转到登录页，不生成脏订单”。
  3. 自动化可行性判定：标注该用例是否适合写进自动化测试脚本（CI/CD），如果是，提供一行伪代码思路（如使用 Selenium 或 Cypress）。
  4. 探索性测试盲区 (Exploratory Testing)：提供 2 个机器很难想到、需要人类 QA 靠直觉去“搞破坏”的测试方向（如断网重连时的状态锁死测试）。
  ```
- **产出物：** 高密度 QA 测试用例表（包含大量极端破坏性场景）、BDD 格式验收标准、探索性测试盲区指引。

## 🟢 步骤四：灰度发布策略与接口集成契约 (Rollout & Integration Contract)
*操作目的：产品做完了不等于能全量上线。必须要定义发布节奏（怎么灰度）和废弃功能的去留，这是从 PM 跨入“产品线负责人”视角的关键。*

- **提示词模板：**
  ```text
  最后，代码即将封板。我们需要规划该版本的上线交付策略以及与外部世界的集成契约。
  
  1. 灰度发布与回滚规则 (Gray Release & Rollback)：
     - 不要一刀切上线！请为本次发布设计一个为期 1-2 周的“灰度放量节奏”（例如：先向 5% 内部白名单开放 -> 观察核心报错率是否超过 X% -> 扩容到 50%）。
     - 定义一键回滚（Rollback）的触发红线（如：支付成功率瞬间跌破 80%）。
  2. API 接口对接契约 (Integration Contract)：
     - 如果本期需求需要调用公司其他团队的接口、或外部第三方系统（如物流查询）。请列出我们的系统必须向外部发送什么数据（Request），以及期盼外部返回什么数据（Response）。
  3. 下期迭代池 (Next Sprint Backlog)：
     - 根据我们在模块八中砍掉的“Should have / Could have”需求，输出一份包含 3-5 个目标的 V2.0 待办列表，确保研发知道后续的演进方向。
  ```
- **产出物：** 灰度与回滚红线策略、跨系统 API 集成约定、V2.0 迭代储备池 (Backlog)。

## 🛑 研发拒收风险与“魔法词汇”终审（阶段 Checkpoint）
*操作目的：PRD 写得再长，只要有一处逻辑没闭环，开发就有理由拒收。*

- **提示词模板（拦截器）：**
  ```text
  PRD 组装完毕。作为技术把关人，在正式发起研发需求评审会 (PRD Review) 之前，请进行最后一轮“防研发拒收”扫雷扫描：
  
  1. 魔法词汇猎杀：全局搜索 PRD 中是否出现了“自动”、“智能”、“相应地”、“根据情况”等典型的产品经理推卸责任的“魔法词汇”？如果发现了，必须立刻指出，并要求用冰冷的逻辑规则替换。
  2. 空中楼阁排查：检查前端 UI 要求展示的所有数据，是否都能在“后端黑盒逻辑”或“API 集成契约”中找到清晰的数据来源接口？如果有没来源的数据，请标红警告。
  3. 确认指令：如果文档毫无歧义、防杠精机制完备，请输出“✅ 工业级交付契约已冻结，准许召开研发启动会，正式进入开发 Sprint。”如果有任何含糊其辞的地方，请打回重铸细节。
  ```

---

### 💡 拆解后的优势：

1.  **破解大模型“注意力耗散 (Laziness)”：** 通过明确指示大模型“先搭骨架（步骤一），再深挖单一模块（步骤二）”，你完美绕开了 AI 偷懒的毛病，能产出一份长达几十页、且处处是细节的真实 PRD。
2.  **强制引入“非功能性需求 (NFR)”：** 产品经理往往只关心功能，但系统崩溃往往是因为没定义 NFR（高并发、毫秒级响应、防重放）。步骤一补齐了这个决定系统稳定性的短板。
3.  **BDD 与自动化测试融合：** 步骤三的测试用例不是传统的流水账，而是基于行为驱动开发（BDD）的 `Given-When-Then` 格式。这种格式开发人员最喜欢，甚至可以直接被 AI 转化为自动化测试代码，极大地拉高了交付物的工程纯度。
